## Project Architecture Rules

- All actions must be represented as CTR dataclasses defined in ctr.py. Executors may only accept CTR objects as input.
- Structured persistent state must use SQLite via db_manager.py. Temporary logs and exported artifacts may use files.
- The NLU pipeline lives in nlu_router.py. All intent classification and parameter extraction must be implemented there.
- New executors must inherit from base_executor.py and implement a run() method. Legacy action functions may remain until refactored.
- No external API calls. Allowed libraries: sentence-transformers (all-MiniLM-L6-v2), sklearn, numpy, pytesseract, pdf2image, PIL, sqlite3, and Python stdlib.
- Python 3.10, Ubuntu VM, no GPU. Do not suggest GPU-dependent solutions.
- Every module exposing functionality must include a run_demo() function demonstrating usage with example CTR objects.
- All filesystem write operations must pass through the checkpoint module for validation and rollback support.

## Response Style

- Write complete, runnable code. No placeholders or pseudocode.
- When modifying existing files, show the full modified file, not a diff.
- After writing code, include 3 test cases verifying the feature works.
- Do not introduce new dependencies unless explicitly allowed in Project Architecture Rules.
